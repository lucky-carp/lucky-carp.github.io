git的简述

git的主要用途

git 是一款开源的分布式版本控制系统，其在软件开发过程中有着很多优势，如：

  1. 版本控制：Git 可以帮助开发人员追踪代码的变化，记录不同版本之间的差异，并能够在需要时恢复到任意一个历史版本。
  2. 分支管理：Git 可以轻松创建和管理分支，开发人员可以在不影响主分支的同时，开展新的实验或开发新功能。
  3. 团队协作：Git 提供了很多工具和指令，可协助团队协作开发，如合并、拉取、提交等操作。
  4. 防止代码丢失：通过将代码存储在 Git 服务器上，可以避免因误操作或硬件损坏等原因导致代码丢失的风险。

总的来说，使用 Git 管理代码可以有效地提高开发人员的工作效率，降低代码管理和协作时的风险，保证代码的可维护性和代码库的稳定性。

git的三种状态

Git 有三种状态，你的文件可能处于其中之一：**已提交(committed)、已修改(modified)和已暂存(staged)**。

  • 已提交表示数据已经安全的保存在本地数据库中。
  • 已修改表示修改了文件，但还没保存到数据库中。
  • 已暂存表示对一个已修改文件的当前版本做了标记，使之包含在下次提交的快照中。

由此引入 Git 项目的三个工作区域的概念：Git 仓库、工作目录以及暂存区域。工作目录、暂存区域以及 Git 仓库如下图所示：

Git 仓库目录是 Git 用来保存项目的元数据和对象数据库的地方。 这是 Git 中最重要的部分，从其它计算机克隆仓库时，拷贝的就是这里的数据。

工作目录是对项目的某个版本独立提取出来的内容。 这些从 Git 仓库的压缩数据库中提取出来的文件，放在磁盘上供你使用或修改。

暂存区域是一个文件，保存了下次将提交的文件列表信息，一般在 Git 仓库目录中。 有时候也被称作‘索引’，不过一般说法还是叫暂存区域。

基本的 Git 工作流程如下：

  1. 在工作目录中修改文件。
  2. 暂存文件，将文件的快照放入暂存区域。
  3. 提交更新，找到暂存区域的文件，将快照永久性存储到 Git 仓库目录。

如果 Git 目录中保存着的特定版本文件，就属于已提交状态。 如果作了修改并已放入暂存区域，就属于已暂存状态。 如果自上次取出后，作了修改但还没有放到暂存区域，就是已修改状态。

Git常用命令

下面是 Git 中常用的一些命令：

  1. git init：初始化一个 Git 仓库。
  2. git add：将文件添加到 Git 的暂存区。
  3. git commit：将暂存区的文件提交到本地仓库。
  4. git clone：将远程仓库克隆到本地电脑。
  5. git push：将本地仓库的代码推送到远程仓库。
  6. git pull：将远程仓库的最新代码拉取到本地电脑。
  7. git branch：列出当前仓库中的所有分支。
  8. git checkout：切换分支。
  9. git merge：将一个分支合并到当前分支中。
  10. git status：查看当前仓库的状态，列出未加入版本控制的文件和已修改的文件。
  11. git log：查看提交历史记录。
  12. git rm：将文件从 Git 仓库中删除。
  13. git stash：将当前的代码修改暂存起来，以便来处理紧急任务。
  14. git rebase ：用于将一个分支的修改内容应用到另一个分支上，或者修改当前分支中的提交历史。

以上仅是 Git 中的部分常用命令。更多的git命令和配置选项可以查看git教程。

使用git向远程仓库提交代码的流程

注：本节以microsim项目为案例进行说明。

  1. 将本地修改提交到本地分支的暂存区。

    • file1 和file2是本地修改过的文件或者是新建的文件
    • git commit -m可以对本次提交内容进行说明

  2. 向develop分支合入代码， 需要先将本地分支checkout到develop分支上，然后拉取远端代码

    • 切换到了develop分支下
    • git pull是将远程分支的修改合并到本地分支。git pull --rebase相当于 git fetch 命令和 git rebase 命令的组合。
    • 它与 git pull 命令的区别在于，git pull 命令相当于执行了 git fetch 和 git merge 两个命令，会将本地仓库和远程仓库合并下来。而 git pull --rebase 命令则是在将远程仓库的代码拉取到本地仓库之后，将当前本地分支和远程分支合并，以避免产生冗余的合并提交。
    • 使用 git pull --rebase 命令的好处是，将远程分支的代码同步到本地分支的时候，避免了出现由于本地分支和远程分支的历史分叉而产生的冲突，从而可以帮助我们更好地维护项目的代码库。

  3. 切换到个人的开发分支，并rebase到develop分支上

    • 切换到我的开发分支dev-liyang-feature-clickhouse上。
    • 将当前分支变基到develop分支上，即将当前分支所在的提交修改为基于 master 分支上的最新提交，并将当前分支上所做的修改逐个应用到新的基础提交之上。
    • 在执行变基操作之前，git会先把当前分支提交保存一份，再将本地分支的提交内容同步到 master 分支最新提交，接着将本地分支的修改逐一以补丁文件（patch）的形式应用到 master 分支上。
    • 使用 git rebase 可以将分叉的提交历史整理成一条直线，以便更好地追溯提交历史和维护分支的合并历史，但是需要注意的是，使用 git rebase 有风险，因为修改提交历史可能会丢失部分提交。

  4. 如何git rebase产生冲突如何处理

    • 当本地分支的提交和目标分支（例如 master 分支）的提交有差异时，git无法确定要保留哪个提交，会导致git rebase中断，并提示未合并的路径，该路径下的文件就是存在冲突的文件，也可以使用git status 查看存在冲突的文件路径。

    • 这时候需要手动解决中断，打开出现冲突的文件，手动编辑文件内容，消除冲突。编辑完成后，需要使用 git add 命令将修改内容添加到暂存区中。vscode提供了更方便的工具，点击未合并的路径会跳转到冲突文件，自己根据需要选择要合并的代码即可，注意记得重新add修改后的文件。

    • 解决完冲突后，需要让rebase继续

  5. 将个人分支推送到远程仓库

以上操作即可将本地修改提交到远程的个人分支。
